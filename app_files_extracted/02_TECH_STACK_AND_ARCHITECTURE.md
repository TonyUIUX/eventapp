## State Management: Riverpod

Use **Riverpod** (not Provider or Bloc). Keep it simple:
- `FutureProvider` for Firebase data fetching
- `StateProvider` for UI state (selected category, toggle)
- `StateNotifierProvider` for saved events list

No complex notifiers unless absolutely necessary.

## Architecture Layers

```
[ Firebase Firestore ]
        ↓
[ FirestoreService ]      ← Raw data access only
        ↓
[ Providers (Riverpod) ]  ← Business logic, filtering, state
        ↓
[ Screens + Widgets ]     ← UI only, no logic
```

**Rules:**
- Screens do NOT call Firebase directly. Always go through a service.
- Providers do NOT have UI code.
- Models are pure Dart classes with `fromFirestore` factory.
- Keep widgets small. If a widget is > 80 lines, extract it.

## Navigation

Use Flutter's built-in `Navigator 2.0` — NOT go_router for MVP.

```dart
// Navigate to detail
Navigator.push(context, MaterialPageRoute(
  builder: (_) => EventDetailScreen(event: event),
));
```

Simple, no complexity needed.

---

## Bottom Navigation

Two tabs:
- Tab 0: Home (event list)
- Tab 1: Saved (bookmarked events)

Use `BottomNavigationBar` with `IndexedStack` to preserve scroll state.

---

## Error Handling Strategy

1. Firebase errors → Show a retry widget (not a dialog)
2. Empty states → Show an illustration + message
3. Image load errors → Show a grey placeholder
4. No internet → Show offline banner (check connectivity via try/catch on Firestore)

Never use `print()` for errors. Use `debugPrint()`.

---

## Performance Rules

- Use `const` constructors everywhere possible
- Use `ListView.builder` (not `.children`) for all lists
- Cache images with `CachedNetworkImage`
- Use `Shimmer` for loading states instead of spinners
- Do NOT load all Firestore documents — query with filters + limit 50
